# Operational Evidence

Status: proposed baseline
Date: 2026-07-01

## Purpose

Operational evidence is the proof that security controls ran, not just that
they were designed. Kotoba should retain evidence as content-addressed artifacts
where possible and index it in EDN.

## Evidence Types

| Evidence | Producer | Required binding |
|---|---|---|
| conformance result | CI/compiler | commit, profile version, command |
| safe-build result | compiler | source CID, policy CID, component CID |
| package verification | registry/installer | manifest CID, tree CID, signer |
| aiueos verify/run | broker | manifest, policy, grant/deny receipt |
| key release | custody | requester DID, purpose, nonce, receipt |
| incident record | operations | timeline, artifacts, owner, closure |
| release security note | security owner | SBOM, provenance, risks, sign-off |
| FIPS boundary evidence | crypto owner | module certificate and policy |

## Evidence Index Schema

Every evidence record should carry:

```edn
{:evidence/id "ev-..."
 :evidence/type :safe-build
 :evidence/created-at "2026-07-01T00:00:00Z"
 :evidence/producer "ci"
 :evidence/artifacts [{:kind :log :digest "sha256:..."}]
 :evidence/claims [:host-imports-policy-derived]
 :evidence/result :pass
 :evidence/retention "P7Y"}
```

## Evidence Quality

Strong evidence:

- generated by the control path itself;
- names command, commit, policy, and artifact digest;
- is immutable or content-addressed;
- can be independently re-run or verified.

Weak evidence:

- screenshots without artifact ids;
- prose claims without command output;
- local-only logs without digest;
- test names that do not cover the claimed control.

## Release Evidence Packet

Each release packet must include:

- evidence index snapshot;
- SBOMs;
- provenance;
- conformance results;
- package verification results;
- unresolved critical/high risk list;
- exception register snapshot;
- key/signature status snapshot;
- release security note.
